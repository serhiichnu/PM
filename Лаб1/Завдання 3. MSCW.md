### ЗАВДАННЯ 3. Зв’язок методу MSCW з життєвим циклом MVP ➔ MMP ➔ MLP

Пріоритезація за методом MSCW дозволяє чітко розмежувати технічні завдання всередині кожного етапу розробки **BookOff**, щоб уникнути **Scope Creep** (розповзання меж). На основі аналізу функціональності за моделлю Кано та її розподілу по стадіях, нижче наведено детальний перелік пріоритетів.

> **Зауваження:** Швидкість інтерфейсу та хмарна синхронізація виступають не як окремі функції, а як нефункціональні вимоги до реалізації Must have модулів, тому вони винесені у технічний аспект стадії MMP.

1. MVP (Minimum Viable Product)
| Пріоритет | Функціональність | Технічний аспект |
| :--- | :--- | :--- |
| **Must have** | Цифрова бібліотека (списки "Хочу/Читаю/Прочитано"). | Схема БД (PostgreSQL) та REST API. |
| **Must have** | Smart-пошук (назва/автор) та реєстрація/авторизація. | Інтеграція з Google Books API та JWT-авторизація. |
| **Must have** | Трекер прогресу (сторінки/відсоток). | CRUD-операції в БД для оновлення полів прогресу. |
| **Should have** | Темна тема інтерфейсу (Dark Mode). | Робота зі стилями у мобільному фреймворку. |
| **Could have** | Базове текстове поле для коментаря. | Збереження одного текстового string у БД. |
| **Won't have** | Сканер штрих-коду (ISBN) та Reading Streak. | Відкладено для комерційної стабілізації продукту в MMP. |



#### 2. MMP (Minimum Marketable Product)
**Мета:** Створення продукту, готового до комерційної конкуренції.

| Пріоритет | Функціональність | Технічний аспект |
| :--- | :--- | :--- |
| **Must have** | Конструктор нотаток із цитатами та тегами. | Розробка текстового редактора з підтримкою метаданих. |
| **Must have** | Reading Streak (щоденне читання). | Логіка перевірки активності через Cron-завдання на бекенді. |
| **Should have** | Плавний інтерфейс і швидка синхронізація. | Впровадження Redis для кешування та оптимізація запитів. |
| **Should have** | Стабільна робота в офлайн-режимі. | Обробка станів Offline-first та механізми синхронізації. |
| **Could have** | Перемикач стилів контенту (beta). | Логіка динамічної підміни текстових ресурсів у додатку. |
| **Won't have** | QA-режим та ігрові фішки (RPG-дерево, дуелі). | Відкладено для формування «Wow-ефекту» на етапі MLP. |

#### 3. MLP (Minimum Loveable Product)
| Пріоритет | Функціональність | Технічний аспект |
| :--- | :--- | :--- |
| **Must have** | Режим стиля контенту. | Динамічна підміна системних повідомлень через промпти. |
| **Must have** | QA-режим (сюжетний Bug Tracker) та Дуелі. | Система репортів з ієрархією Severity та WebSockets. |
| **Should have** | RPG-дерево навичок та "Анти-трекер". | Гейміфікована система лояльності та логіка "покарань". |
| **Could have** | ROI-трекер знань та аналітика. | Складна аналітична логіка порівняння планів та фактів. |
| **Won't have** | Сканер полиць (масове додавання книг). | **Scope Creep** (Відкладено через високу складність Computer Vision). |
| **Won't have** | Повна автономність додатка без користувача. | Вимагає надскладного ШІ (поза межами проєкту). |
