### ЗАВДАННЯ 3. Зв’язок методу MSCW з життєвим циклом MVP ➔ MMP ➔ MLP

[cite_start]Пріоритезація за методом MSCW дозволяє чітко розмежувати технічні завдання всередині кожного етапу розробки **BookOff**, щоб уникнути **Scope Creep** (розповзання меж)[cite: 378, 379]. На основі аналізу функціональності за моделлю Кано та її розподілу по стадіях, нижче наведено детальний перелік пріоритетів.

> [cite_start]**Зауваження:** Швидкість інтерфейсу та хмарна синхронізація виступають не як окремі функції, а як нефункціональні вимоги до реалізації Must have модулів, тому вони винесені у технічний аспект стадії MMP[cite: 267, 271].

#### 1. MVP (Minimum Viable Product)
[cite_start]**Мета:** Довести технічну можливість ведення бібліотеки та базового трекінгу[cite: 373, 382].

| Пріоритет | Функціональність | Технічний аспект |
| :--- | :--- | :--- |
| **Must have** | [cite_start]Цифрова бібліотека (списки "Хочу/Читаю/Прочитано")[cite: 40]. | [cite_start]Схема БД (PostgreSQL) та REST API[cite: 236, 238]. |
| **Must have** | [cite_start]Smart-пошук та реєстрація/авторизація[cite: 41, 43]. | [cite_start]Інтеграція з Google Books API та JWT-авторизація[cite: 232, 278]. |
| **Must have** | Трекер прогресу (сторінки/відсоток). | CRUD-операції в БД для оновлення полів прогресу. |
| **Should have** | Сканування штрих-коду (ISBN). | Використання базових бібліотек для обробки зображень. |
| **Could have** | Темна тема інтерфейсу (Dark Mode). | [cite_start]Робота зі стилями у мобільному фреймворку[cite: 109]. |
| **Won't have** | Конструктор нотаток та Reading Streak. | [cite_start]Відкладено для комерційної стабілізації продукту в MMP[cite: 109]. |



#### 2. MMP (Minimum Marketable Product)
[cite_start]**Мета:** Створення продукту, готового до комерційної конкуренції[cite: 373, 383].

| Пріоритет | Функціональність | Технічний аспект |
| :--- | :--- | :--- |
| **Must have** | Конструктор нотаток із цитатами та тегами. | Розробка текстового редактора з підтримкою метаданих. |
| **Must have** | Reading Streak (щоденне читання). | Логіка перевірки активності через Cron-завдання на бекенді. |
| **Should have** | [cite_start]Плавний інтерфейс і швидка синхронізація[cite: 46]. | [cite_start]Впровадження Redis для кешування та оптимізація запитів[cite: 272]. |
| **Should have** | Стабільна робота в офлайн-режимі. | Обробка станів Offline-first та механізми синхронізації. |
| **Could have** | Перемикач стилів контенту (beta). | Логіка динамічної підміни текстових ресурсів у додатку. |
| **Won't have** | QA-режим та ігрові фішки (RPG-дерево, дуелі). | [cite_start]Відкладено для формування «Wow-ефекту» на етапі MLP[cite: 111]. |

#### 3. MLP (Minimum Loveable Product)
[cite_start]**Мета:** Сформувати лояльність через інновації та емоційний зв'язок[cite: 373, 383].

| Пріоритет | Функціональність | Технічний аспект |
| :--- | :--- | :--- |
| **Must have** | Персоналізація стилів (саркастичний контент). | Динамічна підміна системних повідомлень через промпти. |
| **Must have** | QA-режим ( сюжетний Bug Tracker) та Дуелі. | [cite_start]Система репортів з ієрархією Severity та WebSockets[cite: 113]. |
| **Should have** | RPG-дерево навичок та "Анти-трекер". | Гейміфікована система лояльності та логіка "покарань". |
| **Should have** | Сканер полиць (масове додавання книг). | [cite_start]Computer Vision (TensorFlow/ML Kit) для розпізнавання[cite: 113]. |
| **Could have** | ROI-трекер знань та аналітика. | Складна аналітична логіка порівняння планів та фактів. |
| **Won't have** | Повна автономність додатка без користувача. | [cite_start]Вимагає надскладного ШІ (поза межами проєкту)[cite: 114]. |
