### ЗАВДАННЯ 4. Critical Path Analysis (Аналіз критичного шляху)

**Критичний ризик для MVP: Інтеграція з Google Books API (Smart-пошук)**
Найбільш ризикованою характеристикою для старту є стабільна інтеграція зі стороннім API для пошуку та додавання книг. Оскільки це фундамент "Цифрової бібліотеки", будь-які проблеми з парсингом метаданих, лімітами запитів або відсутністю україномовних видань у базі унеможливлять швидке створення користувачем своїх списків. Без працюючого наповнення бібліотеки етап MVP втрачає свою головну цінність, тому розробка цього модуля є вузьким місцем (Critical Path) всього етапу.

**Критичний ризик для MMP: Безконфліктна хмарна синхронізація (Reading Streak)**
Найбільшим викликом для релізу MMP є забезпечення безперебійної синхронізації прогресу читання між пристроями, особливо після офлайн-сесій. Оскільки на цьому етапі впроваджується "Reading Streak" (трекінг безперервних днів читання), будь-який баг бази даних, який "обнулить" стрік користувача через помилку мережі, викличе сильне роздратування. Технічна складність полягає у правильному вирішенні конфліктів даних (Conflict Resolution), без якого успішний комерційний запуск неможливий.

**Критичний ризик для MLP: Інтеграція LLM для "Режиму стиля контенту"**
Для емоційного релізу MLP найбільш ризикованою є розробка динамічної підміни системних повідомлень через ШІ (іронічно-саркастичний тон). Критичний шлях тут полягає у налаштуванні промптів для LLM-моделі (наприклад, OpenAI API). Якщо відповіді ШІ будуть занадто повільними, галюцинуватимуть або стануть надто образливими, це повністю знищить "Wow-ефект" (Loveable аспект) продукту. Крім того, неконтрольована кількість запитів до API може миттєво виснажити бюджет проєкту.
