### ЗАВДАННЯ 8. Фреймворк SDLC (Software Development Life Cycle)

#### SDLC для стадії MVP

**1. Requirements Engineering (Інженерія вимог)**
* **Процес:** Жорстка фільтрація беклогу. До розробки беруться лише Must-have вимоги (авторизація, пошук книг, базовий трекер сторінок). 
* **Специфіка:** Свідома відмова від гейміфікації на цьому етапі для збереження дедлайнів.

**2. Design (Проєктування)**
* **Процес:** Створення гнучкої, але простої монолітної архітектури.
* **БД:** Проєктування базової реляційної схеми у PostgreSQL (таблиці: `users`, `books`, `user_books_progress`).
* **UI/UX:** Створення базового UI-кіта у Figma (списки книг та екран прогресу).

**3. Implementation / Coding (Розробка)**
* **Процес:** Написання REST API для інтеграції з Google Books API (пошук за назвою).
* **Специфіка:** Розробка кросплатформного мобільного клієнта (напр., на Flutter) з акцентом на швидкість написання коду.

**4. Testing/QA (Тестування та Якість)**
* **Процес:** Фокус на ручному тестуванні (Manual QA) критичних шляхів (Happy Path).
* **Специфіка:** Перевірка, чи коректно зберігаються дані користувача і чи не злітає прогрес прочитаних сторінок при перезапуску додатка.

**5. Deployment / Release (Розгортання)**
* **Процес:** Деплой бекенду на один сервер (VPS) через прості Docker-контейнери.
* **Специфіка:** Публікація бета-версії додатка через TestFlight або Google Play Console для закритої групи тестувальників.

**6. Maintenance (Супроводження)**
* **Процес:** Збір першого фідбеку (чи знаходить пошук україномовні видання?). Моніторинг стабільності сервера.

#### SDLC для стадії MMP

**1. Requirements Engineering (Інженерія вимог)**
* **Процес:** Фокус на нефункціональних вимогах (NFR). Формування логіки обробки втрати інтернету (Offline-first) та правил розрахунку безперервних днів читання (Streaks).

**2. Design (Проєктування)**
* **Процес:** Впровадження механізмів кешування (напр., Redis) для миттєвого відображення статистики.
* **Специфіка:** Проєктування алгоритмів вирішення конфліктів при хмарній синхронізації (якщо юзер читав з двох різних пристроїв офлайн).

**3. Implementation / Coding (Розробка)**
* **Процес:** Реалізація бекенд-джобів (Cron) для щоденного перерахунку Streaks рівно опівночі. Інтеграція SDK для сканування штрих-кодів (ISBN) через камеру.

**4. Testing/QA (Тестування та Якість)**
* **Процес:** Впровадження автоматизованого тестування (Integration Tests). 
* **Специфіка:** Навантажувальне тестування бази даних (симуляція ситуації, коли тисячі користувачів одночасно оновлюють прогрес ввечері).

**5. Deployment / Release (Розгортання)**
* **Процес:** Налаштування повноцінного CI/CD пайплайну (напр., GitHub Actions). 
* **Специфіка:** Розділення середовищ на Staging (для тестування) та Production.

**6. Maintenance (Супроводження)**
* **Процес:** Робота з Crashlytics. Оптимізація "важких" запитів до БД, які формують статистику прочитаного за рік.

#### SDLC для стадії MLP

**1. Requirements Engineering (Інженерія вимог)**
* **Процес:** Глибоке UX-дослідження. Написання промптів та специфікацій для ШІ-помічника (Режим стиля контенту). Формування правил модерації для QA-режиму (Bug Tracker) та ігрової економіки "Дуелей".

**2. Design (Проєктування)**
* **Процес:** Перехід до Event-Driven архітектури (напр., з використанням Kafka/RabbitMQ) для обробки ігрових подій та дуелей у реальному часі.
* **Специфіка:** Проєктування ієрархічної системи збереження баг-репортів для сюжетних дір.

**3. Implementation / Coding (Розробка)**
* **Процес:** Інтеграція LLM API для генерації саркастичних коментарів. Створення логіки QA-режиму для обговорення користувачами помилок у сюжетах. 
* **Специфіка:** Розробка WebSockets для миттєвого оновлення балів під час спільних "Книжкових дуелей". Налаштування ML-моделей (як експеримент для сканера полиць).

**4. Testing/QA (Тестування та Якість)**
* **Процес:** A/B тестування Режиму стиля контенту (перевірка, чи не є жарти додатка занадто образливими).
* **Специфіка:** Тестування системи модерації QA-режиму та перевірка точності експериментального сканера полиць.

**5. Deployment / Release (Розгортання)**
* **Процес:** Використання Canary Releases (нові MLP-фічі спочатку стають доступними лише 5% користувачів).
* **Специфіка:** Оркестрація в Kubernetes (K8s) для автомасштабування серверів під час напливу аудиторії на "Wow-фічі".

**6. Maintenance (Супроводження)**
* **Процес:** Балансування ігрової економіки (RPG-дерева навичок) на основі аналітики поведінки користувачів. Оптимізація витрат на API запити до LLM-моделей.
