Лабораторна робота №3 — BookOff
Завдання 1. Стратегічне, тактичне та операційне планування (SMART)
Проєкт: BookOff — читацький щоденник. Стек: Flutter · Node.js · PostgreSQL · Docker · GitHub Actions.
Стратегічна ціль №1 (SG-1)
Запустити повнофункціональний MVP-додаток BookOff протягом 6 місяців, забезпечивши не менше 100 активних користувачів із core-функціоналом: автентифікація, бібліотека, пошук та трекер прогресу.
Тактичний крок 1.1 — Milestone: Реалізація модуля автентифікації (кінець місяця 1)
SMART-завдання 1.1.1: Розробити та покрити тестами ендпоінти POST /api/auth/register та /api/auth/login на Node.js з bcrypt-хешуванням (saltRounds: 10) та JWT-токенами (exp: 30d) — до кінця Спринту 1 (2 тижні). Критерій: 6 integration-тестів Supertest проходять, час відповіді < 1 с. 
SMART-завдання 1.1.2: Реалізувати Flutter-екрани реєстрації та логіну з валідацією (email RegExp, мін. 8 символів пароля) та збереженням JWT у flutter_secure_storage — до кінця Спринту 1. Критерій: 4 unit-тести форми проходять, токен зберігається між перезапусками.
Тактичний крок 1.2 — Milestone: Реалізація модулів бібліотеки та пошуку (кінець місяця 3)
SMART-завдання 1.2.1: Створити таблицю user_books у PostgreSQL з міграцією Knex та розробити GET /api/user-books?status= — до кінця Спринту 2. Критерій: JOIN з таблицею books повертає метадані, час відповіді < 2 с.
SMART-завдання 1.2.2: Інтегрувати Google Books API через власний проксі GET /api/books/search з Redis-кешуванням (TTL 300 с) — до кінця Спринту 3. Критерій: час відповіді API < 2 с для 95-го перцентиля, cache hit rate > 50% при повторних запитах.
SMART-завдання 1.2.3: Розробити Flutter-екран бібліотеки з TabBar (3 вкладки: Хочу/Читаю/Прочитано) та CRUD-операціями через Riverpod — до кінця Спринту 4. Критерій: Widget-тести покривають всі 3 статуси, дані зберігаються після перезапуску.
Тактичний крок 1.3 — Milestone: Реалізація трекера прогресу та фінальний QA (кінець місяця 5)
SMART-завдання 1.3.1: Реалізувати PATCH /api/user-books/:id з оновленням current_page та автоматичним розрахунком progress_percent на бекенді — до кінця Спринту 5. Критерій: unit-тести покривають граничні значення 0%, 50%, 100%, час відповіді < 1 с.
SMART-завдання 1.3.2: Реалізувати Flutter-екран трекера з LinearProgressIndicator, автоматичним переміщенням книги до «Прочитано» при current_page == total_pages та збереженням дат — до кінця Спринту 5. Критерій: UI-тести покривають всі сценарії, прогрес зберігається після закриття.

Стратегічна ціль №2 (SG-2)
Забезпечити технічну готовність продукту до масштабування на 1000+ користувачів у фазі MMP через стабільну інфраструктуру з uptime Dev-середовища > 99% та автоматизований CI/CD-пайплайн із часом виконання < 5 хвилин.
Тактичний крок 2.1 — Milestone: Налаштування DevOps-інфраструктури (кінець Sprint 0)
SMART-завдання 2.1.1: Налаштувати docker-compose.yml з трьома сервісами (app / db / redis), healthcheck для PostgreSQL та volume для db_data — до завершення Sprint 0 (1 тиждень). Критерій: команда docker-compose up запускає всі сервіси з нуля без помилок протягом 2 хв.
SMART-завдання 2.1.2: Задокументувати та розмістити .env.example з усіма обов'язковими змінними (DATABASE_URL, JWT_SECRET, GOOGLE_BOOKS_API_KEY, REDIS_URL) у репозиторії — до завершення Sprint 0. Критерій: файл .env.example присутній у корені репозиторію з 4 обов'язковими змінними, кожна задокументована коментарем; новий розробник може підняти середовище за README без звернення до команди.
Тактичний крок 2.2 — Milestone: Впровадження CI/CD та стандартів якості (кінець місяця 2)
SMART-завдання 2.2.1: Написати GitHub Actions workflow (.github/workflows/ci.yml) з кроками install → lint → test → build з блокуванням merge при падінні тестів — до кінця Спринту 2. Критерій: PR із failing тестом не може бути змержений; час виконання пайплайну < 5 хв.
SMART-завдання 2.2.2: Налаштувати Swagger-документацію (swagger-jsdoc + swagger-ui-express) для всіх MVP-ендпоінтів (мінімум 8) та зробити її доступною на /api/docs — до кінця Спринту 2. Критерій: 100% ендпоінтів задокументовані, Swagger UI відкривається без помилок.
Тактичний крок 2.3 — Milestone: Перфоманс-тестування та моніторинг (кінець місяця 4)
SMART-завдання 2.3.1: Провести load testing з k6 для ендпоінтів /api/books/search та /api/user-books при 50 одночасних користувачах — до кінця Спринту 4. Критерій: час відповіді для 95-го перцентиля < 2 с, звіт k6 завантажено у Notion.
SMART-завдання 2.3.2: Налаштувати UptimeRobot моніторинг для Dev-середовища з алертами у Slack-канал команди — до кінця Спринту 3. Критерій: uptime > 99% протягом останнього місяця до релізу MVP, підтверджено щотижневим звітом.

Завдання 3. Agile-оцінювання (Story Points & Planning Poker)
3.1 Калібрування — вибір еталонної User Story (Baseline)
Еталонна US: «Як читач, я хочу видалити книгу зі списку» (US №7, Epic 2.0 — Цифрова бібліотека).
Обґрунтування вибору Baseline = 3 SP:
Задача є типовою для проєкту: включає Flutter UI (кнопка + діалог підтвердження), HTTP-запит (DELETE /api/user-books/:id) та оновлення стану через Riverpod — охоплює всі три архітектурні шари мобільного додатку.
Не найпростіша і не найскладніша — знаходиться рівно посередині діапазону складності проєкту, що робить її ідеальною точкою відліку для відносного оцінювання.
Зрозуміла всім учасникам команди без додаткового уточнення вимог: є чіткий Acceptance Criterion і технічний опис у Jira.
Оцінка консенсусна: під час Planning Poker всі учасники одразу погодились на 3 SP — відсутність дискусії підтверджує, що задача добре зрозуміла.
Відносно Baseline = 3 SP: лише UI без API → 2 SP; додається складна бізнес-логіка або зовнішній сервіс → 5–8 SP; системна інфраструктурна задача → 8–13 SP.
